# CRM·비즈니스·휴대폰·메시지 통합 설계

## 1. 목적

CRM을 비즈니스 내부 탭이 아닌 독립된 사이드바 작업 공간으로 제공하고, 제공된 CRM 시안의 정보 구조와 시각 체계를 실제 데이터에 적용한다. 비즈니스에서는 영업·매출, Gmail·Slack 메시지, 명함, 회의 기능에 집중한다. Android와 iPhone은 네이티브 동반 앱을 통해 실제 사용자 계정에 페어링하고 매출 신호, 연락처, 캘린더를 사용자 동의 범위에서 동기화한다.

이 설계는 다음 목표를 하나의 데이터 흐름으로 연결한다.

- 독립 CRM 사이드바 메뉴와 시안 기반 CRM 대시보드
- 실제 Android·iPhone 기기 페어링
- 휴대폰 연락처 가져오기와 CRM 후보 검토
- 휴대폰 캘린더 동기화
- 매출 알림·공유 데이터 수집과 확정 절차
- 명함 이미지 분석과 고객 등록
- 회의 등록과 캘린더·CRM·기억 연결
- Gmail·Slack 대화 목록, 스레드, 답변, 실제 전송
- 비즈니스 탭 간소화
- Polar 결제·구독 관리 버튼

## 2. 범위 분리

구현은 같은 사용자 경험을 유지하되 다음 경계로 분리한다.

1. CRM 작업 공간
2. 비즈니스 작업 공간
3. 모바일 기기 페어링·동기화
4. Gmail·Slack 메시지 작업 공간
5. 명함·회의 입력
6. Polar 결제 진입점

각 경계는 독립 저장소와 API 계약을 갖는다. CRM과 비즈니스 UI가 휴대폰 또는 외부 앱 토큰을 직접 취급하지 않고 서버 API를 통해서만 접근한다.

## 3. 사이드바와 탐색

### 3.1 CRM 독립 메뉴

- `ViewId`와 사이드바 순서에 `crm`을 추가한다.
- CRM 메뉴는 비즈니스와 별도의 `CRMView`를 렌더링한다.
- 현재 `normalizeWorkspaceView("crm")`가 `business`로 바꾸는 호환 처리를 제거한다.
- 새로고침 시 홈이 AI Chat으로 돌아가는 기존 정책은 유지한다.
- 사이드바 순서는 AI Chat, Memory, Business, CRM, Automation, Calendar, Files, Integrations, Settings로 한다.

### 3.2 결제 버튼

결제 버튼은 사이드바 하단의 로컬 저장소 카드 바로 위에 항상 표시한다.

- 미구독자: `결제하기`, Polar Checkout 호출
- 활성 구독자: `결제 관리`, Polar Customer Portal 호출
- 결제 확인 중·연체·취소 상태: 상태에 맞는 안내와 복구 동작
- 관리자 `kara111131@naver.com`: `관리자 무료 이용` 상태를 표시하고 결제를 요구하지 않음
- 요청 중 중복 클릭 차단, 실패 시 한글 오류와 다시 시도

기존 `UpgradeButton`의 숨김·깨진 한글을 수정하고 역할에 맞게 결제 상태 버튼으로 정리한다. Checkout과 Portal URL은 서버에서만 생성한다.

## 4. CRM 디자인과 화면 구조

제공된 시안의 다음 특징을 적용한다.

- 밝은 흰색 중심 배경과 얇은 회색 테두리
- 보라색 주요 동작과 활성 탭
- 작은 둥근 KPI 카드 다섯 개
- 중앙의 단계별 딜 파이프라인 보드
- 오른쪽 활동 요약과 최근 활동
- 하단 최근 연락처 테이블
- 선택한 연락처의 360px 상세 패널
- 높은 정보 밀도를 유지하되 텍스트 넘침이 없는 그리드

### 4.1 CRM 상단

- 제목: CRM
- 보조 설명: 고객 관계와 성장 기회 관리
- 연락처·딜·회사명 통합 검색
- `새 항목` 버튼
- 더보기 메뉴
- 내부 탭: 대시보드, 연락처, 딜, 활동, 이메일, 보고서, 설정

`회사` 탭은 만들지 않는다. 회사명과 직책은 연락처 정보와 딜 정보 안에서 관리한다.

### 4.2 대시보드

KPI:

- 전체 연락처
- 활성 딜
- 예상 수익
- 이메일 오픈율
- 완료 활동

중앙 콘텐츠:

- 단계: 신규, 접촉됨, 제안, 협상, 성사
- 각 단계의 딜 수와 금액
- 드래그로 단계 변경
- 활동 유형 도넛 차트: 이메일, 통화, 미팅, 작업, 메모
- 최근 활동
- 최근 연락처 테이블

### 4.3 연락처 상세 패널

연락처를 선택하면 오른쪽에 다음 내용을 표시한다.

- 프로필 이미지, 이름, 직책, 회사, 중요도
- 전화, 이메일, 미팅, 메모, 더보기 동작
- 탭: 정보, 활동, 딜, 메모, 파일
- 이메일, 휴대전화, 회사 전화, 직책, 회사명, 주소, 웹사이트, 설명
- 태그
- 연결된 딜과 금액·단계
- 담당 소유자

작은 화면에서는 상세 패널을 오른쪽 서랍으로 전환한다.

### 4.4 새 항목

`새 항목`은 다음 진입점을 제공한다.

- 연락처 직접 추가
- 연결된 휴대폰에서 가져오기
- 명함 이미지 분석
- CSV 가져오기
- Gmail·Slack 대화에서 고객 후보 생성
- 딜 추가
- 활동 추가

## 5. 비즈니스 화면

비즈니스 탭은 다음만 유지한다.

- 개요
- 영업·매출
- 메일·메시지
- 명함
- 회의
- 보고서

비즈니스의 `고객`, `회사`, `업무` 탭은 제거한다. 고객 작업은 CRM에서 처리하고, 회의 후속 작업은 CRM 활동 또는 자동화로 연결한다.

### 5.1 영업·매출

- 확정 매출, 예상 매출, 임시 매출 후보, 연결 기기 요약
- `Android 연결`, `iPhone 연결` 버튼
- 연결 기기명, OS, 권한, 마지막 동기화, 상태 표시
- 매출 후보 확인·수정·확정·제외
- CRM 딜과 매출 후보 연결
- 기기 연결 관리로 이동

기존 `Android 자동 수집`·`iPhone 가져오기` 설명 카드를 실제 연결 버튼과 상태 카드로 교체한다.

### 5.2 명함

- `명함 추가` 버튼
- 이미지 업로드, 카메라 촬영, 휴대폰 공유 확장 입력
- 원본 이미지 미리보기
- 이름, 회사, 직책, 전화번호, 이메일, 주소, 웹사이트 추출
- 필드별 신뢰도와 원본 영역 표시
- 중복 연락처 후보 표시
- 사용자가 수정·승인한 경우에만 CRM 연락처 생성
- 원본 이미지와 추출 결과를 사용자 소유 파일로 보존

OCR 또는 AI 분석이 실패해도 원본 이미지와 수동 입력 폼을 유지한다. 원본 명함은 공개 AI 공급자에 자동 전송하지 않으며, 분석 전 사용자 동의와 공급자 정책을 적용한다.

### 5.3 회의

- `회의 추가` 버튼
- 제목, 시작·종료 시각, 참석자, 장소·화상회의 링크
- CRM 연락처·딜 연결
- 안건, 회의 내용, 결정, 후속 작업, 첨부파일
- 캘린더 연결 시 `캘린더 일정 생성` 선택
- 생성 후 CRM 활동과 개인 기억에 연결

회의 저장은 영구 기록 정책을 따르며 UI에서 지워졌다는 이유만으로 메모리 원본을 자동 삭제하지 않는다. 외부 캘린더 생성은 사용자의 명시적 체크와 저장 동작 후에만 실행한다.

## 6. 실제 휴대폰 연결

### 6.1 연결 방식

웹 화면에서 단순히 연결된 것처럼 표시하지 않는다. `mobile-companion` 네이티브 앱과 다음 절차로 페어링한다.

1. 로그인한 웹 사용자가 Android 또는 iPhone 연결 선택
2. 서버가 짧은 유효시간의 단일 사용 QR challenge 생성
3. 동반 앱이 QR 스캔
4. 기기 안에서 비대칭 키 쌍 생성
5. challenge와 기기 공개키를 서버에서 검증
6. 사용자가 기기에서 데이터 종류별 권한 승인
7. 서버가 사용자 소유 기기 레코드와 취소 가능한 세션 생성

웹에는 기기명, OS, 앱 버전, 권한, 마지막 동기화, 연결 상태, 일시정지, 다시 연결, 연결 해제를 표시한다.

### 6.2 수집 범위

| 데이터 | Android | iPhone |
|---|---|---|
| 매출 | 사용자가 선택한 은행·결제 앱 알림 | Share Extension으로 전달한 문구·스크린샷 |
| 연락처 | Contacts 런타임 권한 후 선택 동기화 | Contacts 권한 후 선택 동기화 |
| 캘린더 | Calendar 권한 후 선택 캘린더 동기화 | EventKit 권한 후 선택 캘린더 동기화 |
| SMS·통화기록 | 수집하지 않음 | 수집하지 않음 |

Android 알림 수집은 기본 비활성화하고 사용자가 선택한 패키지만 기기에서 필터링한다. iPhone에서는 다른 앱 알림을 자동 수집한다고 표시하지 않는다.

### 6.3 연락처 동기화

휴대폰 연결 상세에 `연락처 동기화`와 `연락처 추가`를 제공한다.

- 사용자가 전체 또는 선택 연락처 지정
- 이름, 전화번호, 이메일, 회사, 직책, 주소, 사진의 최소 필요 필드만 동기화
- 서버에서 신규, 중복, 충돌, 변경 후보로 분류
- 사용자 확인 후 CRM 연락처로 등록 또는 병합
- `휴대폰에 연락처 추가`에서는 CRM 연락처 또는 새 연락처를 선택하고 대상 기기와 저장 내용을 미리 확인한 뒤 주소록에 생성
- 휴대폰 주소록 쓰기는 별도의 Contacts 쓰기 권한과 사용자의 건별 저장 동작이 있을 때만 실행
- 여러 CRM 연락처를 휴대폰에 자동 일괄 생성하지 않음
- 휴대폰에서 삭제되어도 CRM 연락처 자동 삭제 금지
- 원본 기기 연락처 ID는 암호화된 소유자 범위 매핑에만 사용

### 6.4 캘린더 동기화

- 사용자가 동기화할 캘린더 선택
- 기본은 기기에서 DREAMWISH로 읽기 동기화
- `DREAMWISH에서 일정 수정 허용`을 별도로 켜야 양방향 동기화
- 반복 일정, 시간대, 변경 시각, 삭제 표시를 보존
- 충돌 시 덮어쓰지 않고 사용자에게 양쪽 버전을 표시

### 6.5 매출 후보

- Android 알림 또는 iPhone 공유 데이터는 서명된 암호화 envelope로 업로드
- 기기 ID, 연속 번호, 이벤트 ID, 앱, 시각, 최소 텍스트, parser 버전 포함
- 서버에서 재생 공격, 중복, 시간 범위, 크기, 소유자 검증
- 금액·입출금 방향·상대방·신뢰도를 추출해 임시 후보 생성
- 사용자가 확정하기 전 매출 합계에 포함하지 않음

### 6.6 네이티브 배포 경계

코드베이스에는 Android와 iPhone 실제 페어링·권한·동기화 코드를 구현한다. 실기기 설치용 서명 빌드와 스토어 배포에는 사용자가 소유한 Google Play 및 Apple Developer 계정, 앱 ID, 인증서·프로비저닝 프로필이 필요하다. 자격증명이 없는 상태에서는 개발 서명 빌드와 서버 계약 검증까지 완료하고 가짜 프로덕션 배포를 표시하지 않는다.

## 7. Gmail·Slack 메시지 작업 공간

### 7.1 공통 3열 UI

- 왼쪽: 연결 계정, Gmail 폴더, Slack 워크스페이스·채널
- 중앙: 검색 가능한 이메일 스레드 또는 채널 대화 목록
- 오른쪽: 전체 스레드, 첨부파일, 답장 작성
- 모바일: 계정 → 목록 → 대화의 단계형 화면

동기화 시각, 새로고침, 연결 상태, 재연결 동작을 항상 표시한다.

### 7.2 Gmail

- 받은편지함, 보낸편지함, 임시저장, 별표, 검색
- 스레드별 전체 메시지와 발신자·수신자·시각
- 새 메일, 답장, 전체 답장, 전달, 임시저장
- 첨부파일 메타데이터와 안전한 다운로드
- 사용자가 `전송`을 누르면 미리보기 확인 후 실제 Gmail API 호출
- 필요한 compose·send 범위가 없으면 정확한 재연결 안내

### 7.3 Slack

- 워크스페이스, 공개·허용 채널 목록
- 채널 메시지, 스레드 답글, 작성자, 시각, 반응, 파일
- 새 메시지와 스레드 답글
- 사용자가 `전송`을 누르면 실제 Slack API 호출
- 채널 접근 권한 또는 `chat:write`가 없으면 재연결 안내

### 7.4 데이터 진실성

기존 Gmail·Slack 동기화 저장소는 사용자별 캐시와 검색 인덱스로 유지한다. 화면은 캐시 상태와 마지막 동기화를 명확히 표시하고, 새로고침은 실제 공급자 API에서 다시 가져온다. 전송 성공은 Gmail·Slack이 성공 응답을 반환한 경우에만 표시한다.

## 8. 데이터 모델

주요 레코드는 모두 `ownerId`를 필수로 가진다.

- `paired_devices`: 기기, OS, 공개키, 상태, 마지막 동기화
- `device_permissions`: revenue, contacts, calendar 권한과 갱신 시각
- `device_sync_cursors`: 데이터 종류별 cursor와 시각
- `contact_import_candidates`: 원본 기기 매핑, 정규화 연락처, 중복·충돌 상태
- `calendar_sync_links`: 기기 이벤트와 DREAMWISH 일정의 양방향 매핑
- `business_card_imports`: 원본 파일, 추출 결과, 신뢰도, 검토 상태
- `meeting_records`: 참석자, CRM·딜, 캘린더, 내용, 첨부파일 연결
- `message_sync_state`: Gmail·Slack 계정별 cursor와 마지막 동기화

CRM의 기존 고객·딜·활동 저장소를 유지하되 필요한 필드를 확장한다. 외부 서비스 ID는 사용자 소유 연결 범위에서만 유일해야 한다.

## 10. 보안과 개인정보

- 휴대폰 QR challenge는 단일 사용이며 짧은 시간 후 만료
- 기기 요청은 기기 개인키 서명, timestamp, sequence, idempotency key 검증
- 연결 해제 즉시 새 업로드 차단
- OAuth·기기 Secret과 외부 ID 매핑은 암호화 저장
- 연락처·일정·메시지·명함은 사용자별 격리
- 휴대폰 주소록 전체를 자동 CRM 등록하거나 AI에 자동 전송하지 않음
- SMS·통화기록 수집 금지
- 명함 이미지와 메시지 첨부파일의 MIME, 확장자, 크기 검사
- 전송·외부 일정 생성은 사용자의 명시적 버튼 동작 후 실행
- 실행·전송 로그에 토큰, 쿠키, 메시지 전체 본문을 남기지 않음
- 기기·외부 계정 연결 해제와 데이터 삭제 감사 기록 유지

## 11. 오류 처리

- QR 만료: 새 QR 생성
- 기기 서명·sequence 오류: 업로드 거부와 재페어링 안내
- 권한 거부: 해당 데이터 종류만 비활성화
- 연락처 충돌: 자동 병합 금지, 비교 화면 제공
- 캘린더 충돌: 양쪽 버전 선택
- Gmail·Slack 토큰 만료: 전송 중단과 재연결 안내
- OCR 실패: 원본 유지와 수동 입력
- Polar 실패: 앱 접근 상태를 잘못 활성화하지 않고 다시 시도
- 저장 실패: 작성 중인 입력과 업로드 미리보기 유지

## 12. 반응형과 접근성

- CRM 데스크톱 본문은 `minmax(0, 1fr) + 360px` 상세 패널
- 모든 카드와 그리드 자식에 `min-width: 0`
- 긴 이름·회사·제목은 줄바꿈 또는 말줄임과 전체값 툴팁
- 1024px 미만에서는 상세 패널과 메시지 열을 단계형 서랍으로 전환
- 모든 탭, 목록, 작성, 업로드, 페어링은 키보드 접근 가능
- 상태를 색상만으로 전달하지 않고 텍스트·아이콘을 함께 사용
- 이미지 업로드와 연락처 후보에 명확한 접근 가능한 이름 제공

## 13. 테스트

### 13.1 CRM·비즈니스

- 사이드바 CRM 메뉴와 독립 ViewId
- `crm`이 Business로 변환되지 않음
- 비즈니스의 고객·회사·업무 탭 제거
- CRM 탭, KPI, 파이프라인, 상세 패널
- 연락처·딜·활동 CRUD와 사용자 격리
- 명함 분석 검토와 승인 전 CRM 미등록
- 회의 등록, CRM·캘린더 연결, 메모리 보존

### 13.2 모바일

- 단일 사용 QR, 만료, 서명, replay 차단
- 사용자·기기 격리와 연결 해제
- Android 선택 앱 allowlist
- iPhone Share Extension 입력
- 연락처 신규·중복·충돌·삭제 처리
- CRM 연락처를 건별 승인 후 Android·iPhone 주소록에 추가
- 캘린더 읽기와 선택적 양방향 충돌
- 매출 후보가 승인 전 합계에 포함되지 않음
- SMS·통화기록 권한과 코드가 없음

### 13.3 Gmail·Slack

- 실제 연결 계정만 목록 표시
- Gmail 스레드·Slack 채널과 답글 사용자 격리
- 동기화 cursor와 중복 제거
- 권한 부족·만료·rate limit 오류
- 전송 버튼 전 미리보기
- 공급자 성공 응답 이전 성공 표시 금지

### 13.4 결제와 UI

- 미구독 Checkout, 구독 Portal, 관리자 무료 상태
- 결제 버튼이 로컬 저장소 위에 표시
- 중복 요청 차단과 오류 복구
- 제공 시안과 같은 CRM 레이아웃
- 1024px·1440px에서 글자·패널 넘침 없음
- 새로고침 시 AI Chat 홈 정책 유지

## 14. 구현 순서

이 범위는 다음 순서로 구현하고 각 단계별 테스트를 통과시킨다.

3. Gmail·Slack 3열 읽기 UI와 실제 동기화
4. Gmail·Slack 작성·답장·실제 전송
5. 기기 데이터 모델과 QR 페어링 API
6. Android·iPhone 페어링과 권한 화면
7. 연락처·캘린더 동기화와 후보 검토
8. Android 매출 알림·iPhone 공유 매출 입력
9. Polar 결제 상태 버튼 수정
10. 전체 테스트, 실기기 가능한 범위 검증, 브라우저 검증

## 15. 완료 기준

- 사이드바에 독립 CRM 메뉴가 있고 제공 시안과 같은 대시보드·상세 구조가 실제 CRM 데이터로 동작한다.
- 비즈니스에서 고객·회사·업무 탭이 제거된다.
- 영업·매출에서 Android와 iPhone을 QR로 실제 페어링할 수 있다.
- 휴대폰 연락처를 후보로 확인해 CRM에 추가할 수 있다.
- CRM 연락처 또는 새 연락처를 사용자가 확인한 뒤 연결된 휴대폰 주소록에 추가할 수 있다.
- 휴대폰 캘린더를 읽기 동기화하고 별도 허용 시 양방향으로 동기화할 수 있다.
- Android 선택 알림과 iPhone 공유 데이터가 임시 매출 후보로 들어온다.
- 명함 이미지를 분석하고 사용자가 승인한 경우에만 CRM 연락처로 등록된다.
- 회의를 직접 등록하고 CRM·캘린더·기억에 연결할 수 있다.
- Gmail과 Slack 목록·스레드·답글을 실제 데이터로 확인하고 사용자 전송 동작으로 답변을 보낼 수 있다.
- Polar 결제·구독 관리 버튼이 로컬 저장소 위에서 상태에 맞게 동작한다.
- 관리자 무료 사용, 결제 권한, 사용자 데이터 격리, 비밀값 보호가 유지된다.
- 타입 검사, 관련 테스트, 빌드, 브라우저 검증이 통과한다.
- 구현 변경사항을 `main`에 커밋하고 원격 저장소에 푸시한다.
